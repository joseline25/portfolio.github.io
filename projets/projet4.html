Sous-projet 4 : Réponse à Incident et Analyse Forensique (Blue Team)

Ce projet illustre ma capacité à passer du rôle d'attaquant à celui de défenseur. Il couvre la détection d'activités malveillantes avec un SIEM, l'analyse forensique pour comprendre une compromission, et la mise en œuvre de correctifs pour restaurer et renforcer la sécurité.

Objectif

Détecter des attaques simulées, analyser l'impact, retracer les actions de l'attaquant et appliquer des mesures correctives efficaces. Ce scénario évalue la robustesse des mécanismes de défense et la capacité à répondre rapidement à un incident.

Démarche & Outils

La démarche a été structurée en trois phases : détection (Wazuh), analyse mémoire (Volatility) et application de correctifs.

1. Détection d'Attaques avec le SIEM Wazuh

  • Simulation et détection d’un brute force SSH : attaque lancée contre WebServer1 (outil de test en lab : hydra). L’agent Wazuh a collecté les logs d’authentification (/var/log/auth.log) montrant de multiples tentatives échouées depuis 192.168.45.20.
  • Analyse des alertes : le dashboard Wazuh a affiché en temps réel 703 alertes liées à l’activité suspecte, corrélées avec le framework MITRE ATT&CK (T1110 - Brute Force). Règles déclenchées (exemples) : ID 5760 (échec d'auth SSH), ID 2502 (échecs multiples).
  • Conformité : possibilité de filtrer et d’aligner les événements avec des standards (ex. PCI-DSS).

2. Analyse Mémoire (Forensic) avec Volatility

  • Dump mémoire : création d’un dump mémoire sur WebServer2 pour analyse post-mortem.
  • Investigation : utilisation de Volatility (ex. linux.pslist, linux.sockstat, linux.lsof) : découverte de processus (vsftpd, sshd, apache2), connexions actives vers 192.168.1.101 (ports 21 & 22), et fichiers ouverts suggérant des transferts.

3. Application des Correctifs et Durcissement

Actions réalisées suite à la détection :

  • WebServer1 : analyse des logs Apache, déploiement d’un WAF (ModSecurity), correction du code vulnérable (requêtes préparées).
  • WebServer2 : désactivation de l'accès FTP anonyme (anonymous_enable=NO dans vsftpd.conf).
  • Windows 7 : désactivation SMBv1 et arrêt du service Print Spooler.
  • Réseau : blocage de l’IP attaquante via pfSense/iptables, déploiement de Fail2Ban et durcissement SSH (PermitRootLogin no).

Résultats et Livrables

  • Captures d’écran Wazuh : détection en temps réel, classification MITRE ATT&CK, règles déclenchées.
  • Rapport forensique : sorties Volatility détaillant processus, connexions et fichiers suspects.
  • Liste de remédiations : commandes et configurations appliquées (Apache, VSFTPD, Windows, iptables, Fail2Ban).

Exemples de commandes défensives

Commandes/directions utiles pour la remédiation (administrateur autorisé) :

# Exemple : désactiver l'accès FTP anonyme (vsftpd.conf)
# Dans /etc/vsftpd.conf
anonymous_enable=NO

# Exemple : activer et configurer Fail2Ban (basique)
sudo apt install fail2ban
# ensuite éditer /etc/fail2ban/jail.local pour activer la jail ssh

# Exemple : blocage d'une IP suspecte (iptables)
sudo iptables -A INPUT -s 192.168.45.20 -j DROP

Ces commandes sont destinées aux équipes de défense pour la remédiation et le durcissement.
